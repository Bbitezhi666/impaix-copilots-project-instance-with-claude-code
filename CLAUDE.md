# impAIx - L&D AI Copilot 系统

## 🎯 项目概述

impAIx 是面向企业L&D专业人员的AI Agent协作系统，专门用于培训需求调研工作流。本系统基于经典的L&D专业方法论（ADDIE模型、Goldstein培训分析法、Gilbert六盒模型），通过多Agent协作实现从业务问题识别到解决方案建议的完整分析流程。

## 🏗️ 系统架构

### 核心设计原则
- **专业方法论驱动**: 严格遵循L&D领域最佳实践
- **工作流状态管理**: 基于7节点流程的状态机制
- **上下文智能传递**: 确保分析结果的连续性和一致性
- **质量多层保障**: 每个环节都有专业性验证

### Agent协作体系
- **主Agent**: 工作流编排、任务路由、质量控制
- **专业子Agent**: 6个专家团队处理特定分析任务
- **方法论顾问**: 跨节点专业指导和质量检查

## 📋 工作流程控制

### 7节点培训需求调研流程

**节点1: 开始 (Start)**
- 收集：business_problem, target_department, stakeholder_request
- 输出：结构化问题描述文档

**节点2: 分析计划生成 (Analysis Plan Generator)**  
- Agent: training-needs-analyst
- 方法论：Goldstein三层分析（组织-任务-个人）
- 输出：专业分析行动计划

**节点3: 数据收集工具包生成 (Data Collection Toolkit Generator)**
- Agent: data-collection-designer  
- 输出：分层访谈提纲、调研问卷草稿

**节点4: 原始数据整合 (Human-in-the-Loop)**
- 类型：人工介入点
- 收集：访谈记录、问卷结果、观察笔记、其他数据
- 功能：支持工作流暂停和恢复

**节点5: 信息整合与主题提炼 (Insight Synthesizer)**
- Agent: insight-synthesizer
- 输出：3-5个核心主题，支持证据，矛盾识别

**节点6: 根本原因诊断 (Root Cause Analyzer)**
- Agent: root-cause-analyzer
- 方法论：Gilbert六盒模型
- 输出：系统性根本原因诊断报告

**节点7: 报告与建议生成 (Report & Recommendation Generator)**
- Agent: report-generator
- 输出：完整专业分析报告，分类建议（培训vs非培训）

## 🔧 任务路由规则

### 主Agent职责
1. **工作流编排**
   - 维护节点状态机制
   - 管理workflow-data/目录下的所有状态文件
   - 确保节点间依赖关系正确执行

2. **任务分派**
   - 根据当前节点调用相应的专业Agent
   - 传递完整上下文信息给子Agent
   - 收集并验证子Agent的输出结果

3. **质量控制**
   - 每个节点完成后进行L&D方法论合规性检查
   - 确保输出内容的专业性和可解释性
   - 在关键节点要求用户确认后继续

4. **用户交互**
   - 在节点4提供明确的数据收集指导
   - 在节点切换时展示进度和下一步计划
   - 支持用户在任意节点要求修改或重做

### 状态管理机制

```
workflow-data/
├── session.json          # 当前工作流状态
├── business-problem.md   # 节点1输出：初始问题描述
├── analysis-plan.md      # 节点2输出：分析行动计划
├── data-collection-tools.md # 节点3输出：调研工具包
├── raw-data.md          # 节点4输出：用户输入的原始数据
├── insights.md          # 节点5输出：洞察主题分析
├── root-cause-analysis.md # 节点6输出：根本原因诊断
└── final-report.md      # 节点7输出：最终分析报告
```

### 上下文传递规范

**向子Agent传递信息时，必须包含:**
1. 当前节点编号和名称
2. 前置节点的所有输出结果
3. 原始业务问题的完整描述
4. 适用的L&D方法论框架要求
5. 输出格式和质量标准

**子Agent完成任务后，必须:**
1. 更新对应的状态文件
2. 向主Agent报告完成状态
3. 提供结果的专业依据说明
4. 标识需要用户确认的关键决策点

## 🎪 使用指导

### 启动工作流
用户可以通过以下方式开始培训需求调研：
- 直接描述业务问题，系统自动识别并引导
- 使用 `/needs-analysis:start` 命令启动标准流程
- 从历史项目恢复未完成的分析

### 节点跳转和恢复
- 支持从任意节点恢复分析
- 允许用户修改前置节点的结果
- 提供节点状态的可视化展示

### 质量保障
- 每个分析结论都有方法论依据
- 关键发现都能追溯到原始数据
- 建议方案符合L&D专业实践标准

## ⚙️ 扩展配置

### 自定义方法论
可以通过配置 `ld-methodology-advisor` Agent来添加企业特定的分析框架：
- 自定义评估维度
- 特定行业的最佳实践
- 企业内部的成功案例模板

### 错误处理
- 节点执行失败时自动重试机制
- 数据不完整时的智能提示
- 与企业其他系统集成时的异常处理

## 📊 预期效果验证

本配置将验证POC的6个核心效果：

1. ✅ **Agent基于目标进行自主计划**: training-needs-analyst会根据业务问题自动生成Goldstein三层分析计划
2. ✅ **基于后台工作流展示计划**: 主Agent实时显示7节点进度和当前状态
3. ✅ **基于计划逐步执行任务**: 严格按节点顺序，每个节点完成后自动进入下一节点  
4. ✅ **等待用户反馈后重新规划**: 节点4人工介入，支持基于用户输入调整后续分析
5. ✅ **画布展现各类交付物**: workflow-data/目录下生成8个结构化分析文档
6. ✅ **点击交付物要求调整**: 通过命令系统支持修改任意节点的输出
